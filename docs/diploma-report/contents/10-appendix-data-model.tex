\subsection{Ключевые ограничения и идентификаторы}

Наиболее важные ограничения реляционной модели:
\begin{enumerate}
    \item уникальность события фасада по client\_id + idempotency\_key;
    \item уникальность запуска доставки по event\_id + idempotency\_key;
    \item уникальность фактической доставки по комбинации запуска, получателя и канала;
    \item уникальность записей о попытках по номеру попытки для конкретной доставки;
    \item идемпотентность записи истории по outbox\_id.
\end{enumerate}

Эти ограничения закрывают два класса риска: появление дублей при сетевом повторе
и нарушение согласованности при конкурентной обработке нескольких batch-потоков.

\subsection{Служебные журналы}

С точки зрения архитектуры хранения особое значение имеют две группы таблиц:
\begin{enumerate}
    \item outbox\_message во фасаде и sender-сервисах;
    \item consumer\_inbox\_message в sender-сервисах.
\end{enumerate}

Таблицы outbox содержат:
\begin{enumerate}
    \item тип агрегата;
    \item идентификатор агрегата;
    \item тип события;
    \item payload и headers в JSON-виде;
    \item время создания;
    \item признаки аренды, повтора и подтверждения публикации.
\end{enumerate}

Таблицы inbox содержат:
\begin{enumerate}
    \item идентификатор входящего сообщения;
    \item тип агрегата и тип события;
    \item полезную нагрузку;
    \item статус обработки (NEW, PROCESSING, PROCESSED, FAILED);
    \item время приёма и время завершения.
\end{enumerate}

\subsection{Связь между уровнями данных}

Архитектура хранения выстроена так, чтобы каждый уровень решал собственную задачу:
\begin{enumerate}
    \item PostgreSQL хранит устойчивое доменное состояние, журналы и историю;
    \item Redis хранит оперативный профиль получателя;
    \item Kafka переносит состояние между сервисами в виде событийных сообщений.
\end{enumerate}

Такое распределение ролей позволяет не смешивать транзакционные данные, кэш
получателя и транспорт межсервисного обмена в одном технологическом слое.
