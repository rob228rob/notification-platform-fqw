\section{Проектирование архитектуры распределённой платформы}

\subsection{Архитектурные принципы}

Архитектура платформы строится как набор автономных микросервисов с разделением ответственности по этапам жизненного цикла уведомления \cite{newman_microservices,richardson_patterns,kleppmann_ddia}. Основная граница проходит между командным контуром, который принимает и фиксирует уведомительное событие, и фоновым контуром, который выполняет запуск доставки, канальную обработку и запись истории.

В качестве базовых принципов приняты:
\begin{enumerate}
    \item разделение командного и асинхронного контуров;
    \item локальная транзакционность внутри одного сервиса без распределённых транзакций;
    \item сквозная идемпотентность на уровне команд, запусков доставки и событийного обмена;
    \item сохранение промежуточных состояний в собственных хранилищах сервисных контуров;
    \item централизованная проверка профиля получателя;
    \item наблюдаемость всех основных переходов обработки.
\end{enumerate}

Командный контур использует gRPC и protobuf-контракты для операций, результат которых нужен вызывающей системе сразу: создание уведомительного события, чтение состояния, запуск доставки, чтение истории. Фоновый контур использует Kafka, чтобы вынести фактическую доставку из критического пути клиентского запроса и отделить приём команд от работы sender-сервисов \cite{grpc_docs,kafka_docs,nygard_releaseit}.

Внешние границы платформы и её интеграции показаны на рисунке~\ref{fig:arch-system-context-structurizr}.

\begin{figure}[H]
    \centering

    \caption{Системный контекст платформы уведомлений}
    \label{fig:arch-system-context-structurizr}
\end{figure}

Клиентская система взаимодействует только с командным контуром, а внешние провайдеры доступны только sender-сервисам. Это исключает прямую связность пользовательского запроса с внешним каналом связи и локализует риски деградации на стороне фоновой обработки, снимая с клиента ответственность за успешное завершение доставки.

\subsection{Контейнерная декомпозиция и границы ответственности}

Каждый сервис, владеющий собственным состоянием, использует отдельное хранилище и не обращается напрямую к данным другого сервиса. Межсервисный обмен выполняется через gRPC и Kafka, а не через прямой доступ к базе данных смежного сервиса.

\begin{figure}[H]
    \centering
    \includegraphics[width=\textwidth]{inc/arch-090526/0905arch_ContainersDataOwnership.png}
    \caption{Контейнерная архитектура платформы уведомлений с разделением хранилищ по сервисным контурам}
    \label{fig:arch-containers-owned-storage}
\end{figure}

Notification Facade, Delivery Dispatcher и Profile Consent используют отдельные PostgreSQL-хранилища. History Writer использует ClickHouse как отдельное аналитическое хранилище истории доставки. Template Registry владеет MongoDB-хранилищем шаблонов, Cancellation Service владеет Redis-хранилищем отметок отмены, а sender-сервисы используют собственные Redis-контуры для технической дедупликации команд. Такое разделение исключает превращение базы данных в общий интеграционный слой и принуждает сервисы обмениваться данными только через protobuf контракты или Kafka-сообщения.

\subsection{Верхнеуровневая модель данных и сервисных границ}

Архитектурная модель опирается на четыре уровня сущностей:
\begin{enumerate}
    \item уведомительное событие как описание намерения;
    \item запуск доставки как экземпляр исполнения события;
    \item канальная доставка как обработка пары «получатель --- канал»;
    \item история доставки как агрегированное представление итогов.
\end{enumerate}

Фасад владеет событием и запуском доставки, sender-сервисы владеют канальной доставкой и попытками отправки, history-writer владеет фактами изменения статуса. Template Registry и Profile Consent вынесены в отдельные контуры, поскольку они обслуживают сразу несколько этапов обработки и несколько каналов.

\subsection{Верхнеуровневые потоки данных}

\subsubsection{Поток приёма команды и регистрации события}

Приём пользовательской команды выполняется в синхронном контуре. Notification Facade валидирует запрос, проверяет ключ идемпотентности, сохраняет уведомительное событие, аудиторию и запись outbox. Только после этого событие публикуется в Kafka для дальнейшей обработки.
Для сценария с уже известной аудиторией в запросе может быть запущена транзакционная публикация в Kafka без outbox записи в базу данных.
Сценарий взаимодействий показан на рисунке~\ref{fig:arch-create-event-flow-structurizr}.

\begin{figure}[H]
    \centering
    \includegraphics[width=\textwidth]{inc/arch-090526/0905arch_InboundCommandFlow.png}
    \caption{Входной контур платформы}
    \label{fig:arch-create-event-flow-structurizr}
\end{figure}

Клиент взаимодействует только с Notification Facade. При необходимости фасад обращается к Template Registry, затем сохраняет событие, аудиторию и запись outbox в собственном PostgreSQL-хранилище и передаёт дальнейшую обработку в Kafka. Такое разделение убирает фактическую доставку из критического пути клиентского запроса и фиксирует границу между командным и фоновым контурами системы.

\subsubsection{Поток запуска доставки}

Запуск доставки отделён от этапа приёма команды. Такое решение даёт три практических эффекта:
\begin{enumerate}
    \item запуск можно выполнить сразу или по расписанию без изменения самой модели события;
    \item повторный запуск допустим без пересоздания уведомительного события;
    \item сбой канального контура не влияет на уже завершённый клиентский запрос.
\end{enumerate}

\begin{figure}[H]
    \centering
    \includegraphics[width=0.92\textwidth]{inc/arch-090526/0905arch_ServicesPresentation.png}
    \caption{Поток запуска доставки и прохождения уведомления через канальный контур}
    \label{fig:arch-processing-flow-presentation}
\end{figure}

Рисунок~\ref{fig:arch-processing-flow-presentation} фиксирует упрощённый путь немедленного запуска доставки. После приёма команды фасад не удерживает дальнейшую обработку внутри пользовательского вызова, а переводит её в событийный контур. Далее сообщение проходит через sender-сервис и завершается записью результата в истории. Такой маршрут подчёркивает, что запуск доставки является самостоятельной стадией обработки, а не продолжением синхронного вызова.

\subsubsection{Поток отложенной доставки}

Отложенная доставка вынесена в самостоятельный контур Delivery Dispatcher. Фасад публикует запрос на запуск в общий топик delivery.dispatcher, диспетчер сохраняет задачу в собственном PostgreSQL-хранилище и после наступления планового времени переводит её в рабочий канальный Kafka-топик. Хранение отложенных задач в отдельном сервисе исключает зависимость от таймеров в памяти конкретного экземпляра приложения и одновременно сохраняет единое место принятия решений по маршрутизации.

\begin{figure}[H]
    \centering
    \includegraphics[width=0.92\textwidth]{inc/arch-090526/0905arch_DispatchRoutingFlow.png}
    \caption{Поток отложенной доставки с отдельным контуром планировщика}
    \label{fig:arch-scheduling-flow-presentation}
\end{figure}

Между фасадом и sender-сервисом появляется отдельная стадия маршрутизации. Она принимает событие в Kafka, удерживает задачу до планового времени и только затем публикует рабочий запуск в канал доставки. За счёт этого отложенная отправка не зависит от локальных таймеров в памяти и не требует удерживать долгоживущий пользовательский запрос.

\subsubsection{Поток канальной обработки и истории}

Mail Sender и SMS Sender потребляют свои рабочие топики независимо друг от друга. Перед фактической отправкой sender-сервис выполняет только следующие проверки: локальная дедупликация в Redis, проверку отмены через cancellation-service, проверку deadline, timeout и попытку отправки через выбранный канал. После завершения попытки sender публикует статусное событие, которое history-writer записывает в ClickHouse. При окончательной ошибке или исчерпании лимита повторов выполняется fallback-маршрутизация, при которой в Kafka публикуется отдельное событие в топик delivery.fallback, а delivery-dispatcher формирует новый запуск доставки для альтернативного канала. Такой маршрут отделяет повторную попытку в исходном канале от смены канала связи и не смешивает sender-обработку с оркестрацией нового запуска доставки.

\begin{figure}[H]
    \centering
    \includegraphics[width=\textwidth]{inc/arch-090526/0905arch_EmailNotificationProcessingFlow.png}
    \caption{Поток обработки email-уведомления в sender-контуре}
    \label{fig:arch-delivery-handlers-structurizr}
\end{figure}

На канальном уровне сервис отправитель получает команду на отправку из Kafka, выполняет локальную дедупликацию в Redis, проверяет отмену через cancellation-service, обращается к внешнему провайдеру и завершает обработку публикацией события в сервис истории или публикует fallback-событие для повторной обработки другим каналом.

\subsection{Статусная модель доставки}

Статусная модель отражает правила перехода между этапами обработки и разделяет разные состояния обработки.

\begin{figure}[H]
    \centering
    \includegraphics[width=0.55\textwidth]{inc/statuses-protobuf.png}
    \caption{Общий контракт статусов для всей системы}
    \label{fig:arch-delivery-status-protobuf}
\end{figure}

Состояния DRAFT и SCHEDULED относятся к подготовке уведомительного события и запуску доставки. Они фиксируют, создан ли маршрут доставки и должен ли он быть удержан до планового времени. Состояние PENDING открывает канальный контур, когда запуск доставки уже сформирован, но фактическая отправка по каналу ещё не началась. В целевом варианте эта модель закрепляется в общем protobuf-контракте `notification.common.v1.DeliveryStatus`, чтобы фасадный, sender- и исторический контуры одинаково трактовали переходы между стадиями обработки.

После перехода в SENDING sender-сервис выполняет cancellation-check и попытку отправки через выбранный канал. Если доставка отменена, обработка завершается статусом CANCELED. Если провайдер подтверждает отправку, доставка получает статус SENT. Временная ошибка переводит доставку в RETRY и допускает повторную попытку в пределах заданного лимита. Исчерпание лимита повторов или окончательная ошибка не обязательно ведут сразу к FAILED. Sender-сервис сначала публикует fallback-событие со статусом CHANNEL\_CHANGED, а delivery-dispatcher проверяет наличие альтернативного канала, допустимость fallback-маршрутизации и число уже выполненных смен канала.

Если смена канала допустима, delivery-dispatcher создаёт новый запуск доставки. Число смен канала ограничивается, чтобы исключить циклическую маршрутизацию между email и SMS. Если fallback-маршрутизация не разрешена или лимит смен канала уже исчерпан, фиксируется итоговый статус FAILED.

\subsection{Использование ClickHouse для хранения истории доставки}

Для хранения истории доставки в актуальной версии используется ClickHouse. В отличие от транзакционных контуров, история доставки представляет собой преимущественно append-only набор данных. Новые статусные события добавляются по мере обработки уведомлений, а основные запросы выполняются по клиенту, событию, получателю, каналу, статусу и временному интервалу. Поэтому history-writer рассматривается как владелец аналитического хранилища истории, а ClickHouse используется для оптимизации чтений от внешних и внутренних запросов к истории доставки.

В ClickHouse сохраняется расширенный набор атрибутов доставки: идентификаторы клиента, события, запуска доставки, получателя и канальной доставки, канал, статус, причина ошибки, номер попытки, временные метки, destination, ключ идемпотентности, метаинформация Kafka, а также сериализованные payload и headers. Если статусное событие содержит дополнительные технические атрибуты, например correlation identifier или provider-specific поля, они также сохраняются в аналитической записи.

\subsubsection{Маршрут событийного потока}

Маршрут обработки в Kafka-контуре может быть представлен как последовательность доменных шагов:
\begin{enumerate}
    \item фасад публикует событие жизненного цикла уведомления в notification.facade.events;
    \item фасад публикует dispatch-request в общий топик delivery.dispatcher;
    \item delivery-dispatcher выбирает канал, а при отложенной отправке удерживает задачу до планового времени и затем публикует рабочий запуск в notification.mail.dispatches или notification.sms.dispatches;
    \item канальные обработчики публикуют итог в notification.mail.delivery-statuses или notification.sms.delivery-statuses;
    \item сервис истории консолидирует статусные события в агрегированное представление получателя.
\end{enumerate}

\subsection{Механизмы надёжности на архитектурном уровне}

Надёжность обеспечивается сочетанием архитектурных механизмов:
\begin{enumerate}
    \item transactional outbox в транзакционных контурах, где локальное изменение состояния должно быть согласовано с публикацией интеграционного события;
    \item дедупликация через Redis в sender-сервисах для подавления повторной обработки канальных команд;
    \item ключ идемпотентности и уникальные ограничения для повторных входных команд;
    \item повторная попытка для временных ошибок и статусная модель для окончательных ошибок;
    \item history-writer как контур накопления фактов изменения статуса.
\end{enumerate}

Эта схема соответствует модели at-least-once доставки и управляемой eventual consistency между сервисами \cite{kafka_docs,richardson_patterns}. Дубликаты в транспортном слое допускаются, но подавляются на уровне приложения ключами идемпотентности, дедупликацией через Redis и уникальными ограничениями.

\subsection{Отказоустойчивость}

В работе под отказоустойчивостью понимается не физическая гарантия получения сообщения конечным пользователем, а способность платформы сохранить принятое уведомительное событие, продолжить обработку после частичных сбоев и зафиксировать итог маршрута в истории доставки. Рассматриваются падение экземпляра сервиса, повторная доставка Kafka-сообщения, timeout внешнего провайдера, временная недоступность profile-consent или template-registry, а также отмена во время обработки.

Во входном контуре notification-facade сохраняет доменное изменение и outbox-запись в одном локальном транзакционном шаге. Поэтому сбой отдельного экземпляра сервиса после фиксации команды не приводит к потере уже принятого уведомительного события. В канальных контурах sender-сервисы используют дедупликацию через Redis и устойчивые ключи доставки. Повторная доставка Kafka-сообщения допускается, но не должна приводить к повторному вызову внешнего провайдера для уже обработанной команды.

Ошибки внешнего провайдера не трактуются как единый тип отказа. Временная ошибка переводит доставку в RETRY и допускает повторную попытку в пределах лимита. Окончательная ошибка или исчерпание лимита повторов могут завершить маршрут статусом FAILED, а при наличии разрешённого альтернативного канала инициируют fallback-маршрутизацию. Mail sender публикует событие о канальном неуспехе в delivery.fallback, после чего delivery-dispatcher формирует новый запуск доставки в канал из списка fallback\_channels. Для защиты от циклов используется массив previous\_channels, fallback\_depth и ограничение числа смен канала.

Отметки отмены в Redis покрывают сценарий отмены во время обработки: sender проверяет cancellation-service перед фактической отправкой. Если физическая отправка уже завершилась, отмена не меняет результат внешнего провайдера, но останавливает ещё не обработанные или повторно полученные команды.

Отдельной частью отказоустойчивости выступает контроль деградации на стенде. Readiness- и liveness-probes позволяют Kubernetes вывести из маршрута нездоровый экземпляр сервиса, а метрики gRPC, Kafka producer/listener, consumer lag и доли ошибок дают возможность обнаружить зависание публикации, рост очереди или деградацию sender-контура \cite{kubernetes_in_action}. Локальный k3d-стенд подтверждает прохождение основных сценариев, но не является доказательством production capacity.

\subsection{Архитектура профиля получателя}

Профиль получателя выделен в отдельный сервисный контур. Profile Consent является единственной точкой доступа к PostgreSQL-хранилищу профилей получателей, согласий по каналам и tenant-aware destination. Канальные сервисы и dispatcher не обращаются к таблицам напрямую, а получают решение о допустимости доставки через сервисный контракт profile-consent.

На архитектурном уровне профиль включает:
\begin{enumerate}
    \item активность получателя;
    \item разрешения и ограничения по каналам;
    \item адресные данные, необходимые для email и SMS;
    \item технические атрибуты актуальности профиля.
\end{enumerate}

\subsection{Архитектура шаблонного контура}

Template Registry является единственным владельцем MongoDB-хранилища шаблонов. Остальные сервисы не обращаются к MongoDB напрямую и получают шаблонное содержимое только через сервисный контракт template-registry.

Выбор документоориентированной модели обусловлен структурой предметной сущности. Шаблон включает метаданные, активную версию и несколько канальных представлений, которые в типовом сценарии читаются совместно. Хранение этих данных в одном документе уменьшает число операций чтения при RenderPreview и при выборе активной версии для отправки.

\subsection{Архитектура наблюдаемости}

Наблюдаемость проектируется как обязательная часть архитектуры. Каждый сервис экспортирует метрики, достаточные для восстановления маршрута от входной команды до записи статуса в историю. Наиболее значимыми для этой платформы являются четыре группы метрик:
\begin{enumerate}
    \item gRPC server metrics командного контура;
    \item Kafka producer metrics на публикации событий и статусных сообщений;
    \item Kafka listener metrics на канальной обработке и в history-writer;
    \item JVM и process metrics для оценки ресурсных ограничений.
\end{enumerate}

Такой набор нужен не для общего мониторинга инфраструктуры, а для диагностики разрыва маршрута обработки: принят ли запрос, опубликовано ли событие, дошло ли оно до sender-сервиса и записан ли итоговый статус \cite{spring_boot_docs,kafka_docs}.

\subsection{Безопасность}

В пределах работы безопасность рассматривается на уровне сервисных границ и владения данными. Прикладной сервис не читает чужое хранилище напрямую, а использует только контракт владельца. Это уменьшает площадь непреднамеренного воздействия, упрощает разграничение ответственности и исключает зависимость бизнес-логики от внутренней схемы чужого хранилища.

Выбранная архитектура задаёт основу реализации: gRPC используется для командного контура, Kafka --- для фоновой обработки, а сохранность и повторяемость обработки обеспечиваются outbox/inbox, идемпотентностью и историей статусов.
