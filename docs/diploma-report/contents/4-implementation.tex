\section{Реализация системы, моделирование данных и контракты взаимодействия}

\subsection{Технологический стек и модульная структура}

Система реализована как набор микросервисов на Java~25 с использованием фреймворка Spring Boot.
На текущем этапе развития системы было принято решение хранить всю кодовую базу в рамках одного репозитория для более быстрой разработки и тестирования.
Формат моно-репозитория позволяет сохранить единый цикл сборки для всех сервисов и одновременно с этим разделить прикладные сервисы,
контрактные библиотеки и вспомогательные модули.

В прикладную часть проекта входят следующие java-сервисы:
\begin{enumerate}
    \item notification-facade;
    \item template-registry;
    \item profile-consent;
    \item scheduler-delivery;
    \item notification-mail-sender;
    \item notification-sms-sender;
    \item history-writer;
    \item custom-loader, используемый для нагрузочных и проверочных сценариев.
\end{enumerate}

Контракты вынесены в отдельные protobuf-модули. Это исключает дублирование сигнатур между
клиентскими и серверными компонентами и упрощает сопровождение межсервисных интерфейсов.

В реализации используется множество библиотек из экосистемы Spring, в том числе это: Spring gRPC, Spring Kafka, Spring Data MongoDB,
Spring Data Redis, JavaMailSender, Micrometer и Actuator
\cite{grpc_docs,spring_boot_docs,mongodb_docs,rogov_postgresql17_internals}.

Контур хранения шаблонов реализован на базе документоориентированного хранилища MongoDB.
Операционные контуры событий, запусков доставки,
канальной обработки, планировщика и истории в основном используют реляционную модель в PostgreSQL.
Профиль получателя и прикладные ограничения по каналам связи хранятся в Redis.

\subsection{Моделирование данных}

\subsubsection{Реляционная модель}

Реляционная часть модели распределена по сервисным контурам.
В фасаде хранятся события уведомления, запуски доставки и outbox события для отправки в асинхронный контур.
В канальных сервисах хранятся входящие сообщения, сущности доставки, попытки отправки и статусные записи.
В сервисе истории хранится нормализованный факт изменения статусов доставки.
В планировщике хранится очередь отложенных запусков.

В целевой архитектуре каждый сервисный контур рассматривается как владелец собственного
набора данных и собственного хранилища. Такое разделение уменьшает связанность между
сервисами, упрощает развитие модели данных в каждом контуре и локализует влияние отказов
или миграционных изменений.

Для борьбы с дублированием на уровне данных используются уникальные ключи по бизнес-идентификаторам.
Это предотвращает повторное создание одинаковых сущностей при повторных вызовах и повторной
доставке сообщений. Для отдельных событий идемпотентность гарантированна на уровне отдельного ключа идемпотентности, который служит неким идентификатором состояния запроса. В рамках системы клиентская часть всегда гарантирует, что данный ключ будет оставаться неизменным при неизменных данных запроса.
Другими словами - запросы не будут различаться во время ретраев, если их ключи идемпотентности идентичны. Данный подход
создает некие дополнительные издержки на хранение и проверку данных индентификаторов, но гарантирует консистентность данных.

\subsubsection{Документоориентированная модель шаблонов}

В сервисе template-registry шаблон хранится в коллекции templates.
Документ включает в себя ряд обязательных полей:
\begin{enumerate}
    \item идентификатор шаблона и клиента (templateId, clientId);
    \item метаданные шаблона (name, description, status);
    \item номер активной версии (activeVersion);
    \item idempotency-ключ создания (createIdempotencyKey);
    \item массив versions, в котором каждая версия содержит engine, version,
    список канальных блоков contents на момент создания.
\end{enumerate}

Для этого контура используются следующие индексы:
\begin{enumerate}
    \item уникальный составной индекс (clientId, templateId);
    \item уникальный sparse-индекс (clientId, createIdempotencyKey);
    \item индекс (clientId, updatedAt) для постраничной выдачи шаблонов.
\end{enumerate}

Такая индексная схема поддерживает быстрый доступ к шаблону и обеспечивает
идемпотентность операции CreateTemplate без изменения внешнего gRPC-контракта
\cite{mongodb_docs}.

Для прикладного контура это уменьшает задержку чтения и упрощает контроль целостности версий на уровне сервиса template-registry.

\subsubsection{Модель профиля получателя}

Профиль получателя хранится в Redis. В нём фиксируются признаки активности,
доступности каналов, адресные атрибуты и метка актуальности профиля.
Внутренний формат хранения изолирован внутри сервиса profile-consent:
остальные сервисы работают только через его контракт и не зависят от структуры ключей,
наименования полей и способа представления данных в Redis.

\subsubsection{Контракты управления событиями и доставкой}

Контракт фасада разделён на три группы операций:
\begin{enumerate}
    \item управление событием уведомления;
    \item управление аудиторией;
    \item управление запуском доставки.
\end{enumerate}

Такое разделение соответствует выбранной предметной модели: создание события и запуск доставки
не объединяются в одну команду, а рассматриваются как разные этапы обработки.

\subsubsection{Контракты шаблонов, профиля и истории}

Сервис шаблонов предоставляет операции управления версиями шаблонов и предварительного рендеринга.
При переходе на MongoDB внешний gRPC-контракт сохранён; изменение затронуло только внутренний слой хранения.
Сервис профиля предоставляет операции одиночной и пакетной проверки получателей.
Сервис истории предоставляет операции чтения истории и агрегированных сводок.

\subsubsection{Событийные контракты}

Для асинхронного обмена используется единая структура событийного сообщения,
включающая метаданные события и полезную нагрузку.
Унификация событийного контракта упрощает подключение новых обработчиков и
облегчает трассировку межсервисного потока.

В текущем потоке используются типовые события EventCreated, DispatchCreated,
MailDispatchRequested, SmsDispatchRequested,
MailDeliveryStatusChanged и SmsDeliveryStatusChanged.
Этого набора достаточно для прохождения пути от регистрации события до фиксации
итогового результата в сервисе истории.

\subsection{Ключевые механизмы реализации}

В реализации зафиксированы механизмы, непосредственно подтверждающие принятые архитектурные решения:
\begin{enumerate}
    \item согласованная фиксация доменного состояния и интеграционных сообщений;
    \item двухэтапная обработка входящих событий в канальных контурах для гарантированной сохранности во время сбоев, регистрация входа и последующая доставка;
    \item явное различение временных и окончательных ошибок доставки;
    \item поддержка немедленного и отложенного сценариев запуска доставки;
    \item предварительный рендеринг шаблонов до канальной отправки при необходимости;
    \item optimistic locking при публикации новой активной версии шаблона.
\end{enumerate}

В данном списке отражены только основные элементы реализации, которые непосредственно влияют
на целостность данных, корректность ретарев и устойчивость
основного потока уведомлений к сбоям и непредвиденным ошибкам. Низкоуровневые детали отдельных классов и методов
доступны к просмотрю в приложении.
