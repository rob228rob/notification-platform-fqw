\section{Реализация системы}

Реализация выполнена как набор сервисов на Java~25 и Spring Boot. Код размещён в Gradle monorepo: прикладные сервисы, protobuf-контракты и вспомогательные библиотеки собираются независимо, но используют общие соглашения по конфигурации, наблюдаемости и интеграции.

Синхронные операции выполняются через gRPC. Фоновая доставка, статусы и fallback-события передаются через Kafka. Для транзакционных данных используются PostgreSQL-хранилища, для шаблонов --- MongoDB, для истории статусов --- ClickHouse. Redis применяется только для TTL-механизмов: отметок отмены и локальной дедупликации sender-команд.

\subsection{notification-facade}

Notification Facade является входным сервисом в систему -- командным фасадом платформы. Он принимает клиентские gRPC-команды, проверяет входные данные, фиксирует уведомительное событие и аудиторию, управляет запуском доставки и публикует запрос на  отправку в Kafka топик delivery.dispatcher. Так же фасад отвечает за роутинг внешних запросов в сервисы template-registry, cancellation-service.

Основное хранилище сервиса --- PostgreSQL. В нём хранятся события со статусами, снапшоты аудитории, внешние запросы на доставку и outbox-записи. Transactional outbox используется для согласования записи события и подготовки публикации: доменное изменение и запись на публикацию фиксируются в одной транзакции, а отдельный поток в рамках того же процесса публикует сообщение в Kafka.

\subsection{template-registry}

Template Registry хранит шаблоны и их версии, выбирает активную версию и выполняет рендеринг по параметрам. Сервис отделяет работу с содержимым уведомления от фасада и sender-сервисов: остальные контуры получают уже подготовленное представление через gRPC.

В реализации используется MongoDB. Документ шаблона содержит templateId, clientId, activeVersion, набор версий, channel, subject/body или payload, список переменных и metadata. Такая структура соответствует предметной сущности шаблона: версия и её канальные представления обычно читаются вместе.

\subsection{profile-consent}

Profile Consent является единственной точкой проверки профиля получателя. Сервис отвечает за активность профиля, наличие контактных данных, разрешённость канала, tenant-aware destination и блокировки. Sender-сервисы и delivery-dispatcher не читают профиль напрямую из базы.

Основное хранилище profile-consent --- PostgreSQL. Модель включает recipient\_profile и recipient\_channel\_consent. Первая таблица хранит clientId, recipientId, активность и базовые настройки профиля. Вторая таблица хранит пару channel-tenant, destination, enabled, blacklisted/blockedUntil и служебные timestamps.

\subsection{delivery-dispatcher}

Delivery Dispatcher заменяет старый узкий планировочный контур. Его задача шире, чем отложенный запуск: сервис потребляет dispatch requests из delivery.dispatcher, хранит состояние планирования, проверяет профиль через profile-consent, выбирает канал, публикует sender commands и обрабатывает fallback-события из delivery.fallback.

Dispatcher не выполняет входную валидацию, не работает с аудиторией и не рендерит шаблоны. Эти операции остаются в notification-facade и template-registry. Зона ответственности dispatcher начинается после появления валидного dispatch request в Kafka.

Основное хранилище dispatcher --- PostgreSQL. В нём хранится dispatch state, scheduled dispatch, routing decision, fallback state и outbox для публикации команд. При fallback-маршрутизации sender передаёт previous\_channels: список каналов, через которые доставка уже пыталась пройти. Dispatcher выбирает следующий канал только из fallback\_channels, исключая все previous\_channels. Если альтернативы нет, он фиксирует финальный статус ошибки через статусный pipeline и не создаёт новый цикл маршрутизации.

\subsection{cancellation-service}

Cancellation Service отвечает за отметки отмены. Фасад вызывает его при клиентском запросе отмены, а sender-сервисы проверяют его перед обращением к внешнему провайдеру. Отмена не гарантирует остановку уже завершённой физической отправки, но позволяет остановить ещё не обработанную или повторную доставку.

Хранилище cancellation-service --- Redis. Отметка отмены хранится как TTL-key, связанный с dispatchId или deliveryId. Время жизни ключа составляет 30 минут. Повторная запись отмены обрабатывается идемпотентно.

\subsection{notification-mail-sender}

Notification Mail Sender потребляет email-команды из Kafka, выполняет локальную дедупликацию через Redis, проверяет cancellation-service и profile-consent, затем вызывает email provider. Если отмена уже зафиксирована, внешний provider API не вызывается, а sender публикует статус CANCELED.

Redis Mail Dedup хранит TTL-ключ mail:dedup:{messageId}. Ключ записывается атомарно через SET NX EX. Это не является бизнес-истиной, а только защищает sender от повторной обработки одной и той же Kafka-команды. При временной ошибке отправки выполняется повторная попытка в пределах лимита. После окончательной ошибки mail sender публикует статус FAILED и fallback-событие в delivery.fallback. В fallback-событие записывается previous\_channels, куда добавляется текущий канал CHANNEL\_EMAIL.

\subsection{notification-sms-sender}

Notification SMS Sender реализует аналогичный execution-контур для SMS. Он потребляет SMS-команды, выполняет дедупликацию через Redis, проверяет cancellation-service и profile-consent, затем вызывает SMS provider API. При отмене публикуется статус CANCELED. При временной ошибке выполняется повторная попытка, а итоговый статус передаётся в Kafka для history-writer.

SMS sender не владеет шаблонами, профилем получателя или правилами fallback-маршрутизации. Эти решения остаются в template-registry, profile-consent и delivery-dispatcher.

\subsection{history-writer}

History Writer вынесен из online-path доставки и работает как потребитель статусных событий из Kafka. Сервис не управляет доставкой и не принимает routing-решений. Его задача --- построить историю для аудита, диагностики и чтения текущего состояния.

Хранилище history-writer --- ClickHouse. Таблица истории имеет append-only характер и содержит eventId, dispatchId, deliveryId, recipientId, channel, status, attempt, reason, createdAt/occurredAt и технические атрибуты Kafka-публикации. На чтение доступны операции получения истории по событию, получателю или deliveryId, а также агрегированные сводки по получателю.
