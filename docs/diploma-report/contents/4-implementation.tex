\section{Реализация системы, моделирование данных и контракты взаимодействия}

\subsection{Технологический стек и модульная структура}

Реализация выполнена как многомодульный Gradle-проект на Java 21 и Spring Boot 3.4.x.
Выбор такого формата обеспечивает единый цикл сборки и одновременно разделяет прикладные сервисы и контрактные библиотеки.

В составе прикладной части реализованы модули:
\begin{enumerate}
    \item notification-facade;
    \item template-registry;
    \item profile-consent;
    \item scheduler-delivery;
    \item notification-mail-sender;
    \item notification-sms-sender;
    \item history-writer;
    \item custom-loader для нагрузочных и проверочных сценариев.
\end{enumerate}

Контракты вынесены в отдельные protobuf-модули, что исключает дублирование сигнатур между клиентскими и серверными компонентами.

В реализации используются Spring gRPC, Spring Kafka, Spring Data MongoDB,
Spring Data Redis, JavaMailSender, Micrometer и Actuator
\cite{grpc_docs,spring_boot_docs,mongodb_docs,redis_docs,micrometer_docs}.

Слой хранения шаблонов реализован на Spring Data MongoDB.
Операционные контуры событий и доставки сохраняют реляционную модель в PostgreSQL,
а template-registry использует документоориентированное хранение для шаблонов и версий.

\subsection{Моделирование данных}

\subsubsection{Реляционная модель}

Реляционная модель разделена по сервисным схемам.
В контуре фасада хранятся события, аудитория, запуски доставки и выходящий интеграционный журнал.
В канальных контурах хранятся входящие сообщения, сущности доставки, попытки отправки и статусный журнал.
В сервисе истории хранится нормализованный факт изменения статусов доставки.
В планировщике хранится очередь отложенных запусков.

Такая структура подтверждает архитектурный принцип раздельной ответственности контуров:
\begin{enumerate}
    \item контур команд отвечает за управление жизненным циклом события;
    \item канальные контуры отвечают за исполнение и повторные попытки;
    \item контур истории отвечает за консолидацию результатов.
\end{enumerate}

На уровне ограничений данных применяются уникальные ключи для подавления дублей по бизнес-идентификаторам.
Это обеспечивает практическую idempotency при повторных вызовах и повторной доставке сообщений.

\subsubsection{Документоориентированная модель шаблонов}

В сервисе template-registry шаблон хранится в одной коллекции `templates`.
Документ включает:
\begin{enumerate}
    \item идентификатор шаблона и клиента (`templateId`, `clientId`);
    \item метаданные шаблона (`name`, `description`, `status`);
    \item номер активной версии (`activeVersion`);
    \item idempotency-ключ создания (`createIdempotencyKey`);
    \item массив `versions`, где каждая версия содержит `engine`, `version`,
    список канальных блоков `contents` и момент создания.
\end{enumerate}

Для этого контура используются индексы:
\begin{enumerate}
    \item уникальный составной индекс `(clientId, templateId)`;
    \item уникальный sparse-индекс `(clientId, createIdempotencyKey)`;
    \item индекс `(clientId, updatedAt)` для постраничной выдачи шаблонов.
\end{enumerate}

Индексная модель поддерживает быстрый доступ к шаблону клиента и контроль
идемпотентного `CreateTemplate` без изменения внешнего gRPC-контракта
\cite{mongodb_docs,mongodb_indexes_docs}.

\subsubsection{Модель профиля получателя}

Профиль получателя хранится в Redis и включает признаки активности, доступности каналов, адресные атрибуты и метку обновления.
Реализация изолирует эту модель внутри сервиса profile-consent: другие сервисы работают только через его контракт и не зависят от внутреннего формата ключей и полей.

\subsection{Контракты взаимодействия}

\subsubsection{Контракты управления событиями и доставкой}

Контракт фасада разделён на три группы операций:
\begin{enumerate}
    \item управление событием уведомления;
    \item управление аудиторией;
    \item управление запуском доставки.
\end{enumerate}

Такое разделение реализует принцип, сформулированный в архитектуре: создание события и запуск доставки не смешиваются в одну команду.

\subsubsection{Контракты шаблонов, профиля и истории}

Сервис шаблонов предоставляет операции управления версиями шаблонов и предварительного рендеринга.
Внешний gRPC-контракт сохранён, изменение затронуло только внутренний persistence-слой.
Сервис профиля предоставляет операции одиночной и пакетной проверки получателей.
Сервис истории предоставляет операции чтения истории и агрегированных сводок.

В совокупности эти контракты закрывают базовые зависимости канальных обработчиков:
подготовку содержимого, проверку допустимости доставки и контроль частоты взаимодействия с получателем.

\subsubsection{Событийные контракты}

Для асинхронного обмена используется единая структура событийного сообщения с метаданными события и полезной нагрузкой.
Унификация контракта снижает сложность подключения новых обработчиков и упрощает трассировку межсервисного потока.

\subsection{Ключевые механизмы реализации}

В реализации зафиксированы механизмы, непосредственно подтверждающие архитектурные решения:
\begin{enumerate}
    \item согласованная фиксация доменного состояния и интеграционных сообщений;
    \item двухэтапная обработка входящих событий в канальных контурах (регистрация входа и последующая прикладная обработка);
    \item явное различение временных и окончательных ошибок доставки;
    \item поддержка немедленного и отложенного сценария запуска доставки;
    \item предварительный рендеринг шаблонов при необходимости до канальной отправки;
    \item optimistic locking при публикации новой активной версии шаблона.
\end{enumerate}

Эти механизмы описаны на уровне поведения системы.
Низкоуровневые детали классов, методов и внутренней служебной логики оставлены в исходном коде и технических приложениях.

\subsection{Ограничения текущей версии}

Текущая версия реализации имеет ограничения, которые необходимо явно фиксировать:
\begin{enumerate}
    \item часть каналов может работать в тестовом режиме без фактического внешнего отправителя;
    \item в демонстрационном контуре отдельные соединения настроены без полноценного защищённого транспорта;
    \item модель профиля уже содержит атрибуты для push-канала, но отдельный производственный push-контур не реализован;
    \item часть событийных веток используется как задел на расширение и может быть не задействована в типовом базовом сценарии.
\end{enumerate}
