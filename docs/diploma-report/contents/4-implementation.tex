\section{Реализация системы, моделирование данных и контракты взаимодействия}

\subsection{Технологический стек и модульная структура}

Реализация выполнена как много-модульный Gradle-проект на Java 21 и Spring Boot
3.4.x. Такой формат позволил разделить прикладные сервисы, protobuf-контракты и
вспомогательные библиотеки, сохранив единый процесс сборки и единые правила
версионирования.

В прикладной части проекта реализованы следующие сервисные модули:
\begin{enumerate}
    \item notification-facade;
    \item template-registry;
    \item profile-consent;
    \item scheduler-delivery;
    \item notification-mail-sender;
    \item notification-sms-sender;
    \item history-writer;
    \item custom-loader как утилита генерации нагрузки и сервисных сценариев.
\end{enumerate}

Отдельными модулями вынесены protobuf-контракты фасада, профиля получателя,
истории и шаблонов. Это позволяет переиспользовать единые .proto-описания в
серверной и клиентской частях без дублирования сигнатур.

С точки зрения прикладных библиотек используются:
\begin{enumerate}
    \item Spring gRPC для серверных и клиентских вызовов;
    \item Spring Kafka для transport-слоя и фоновых консьюмеров;
    \item Spring JDBC для доступа к PostgreSQL;
    \item Flyway для миграций схем данных;
    \item Spring Data Redis для чтения профилей коммуникаций;
    \item JavaMailSender для почтового адаптера;
    \item Micrometer и Actuator для метрик и диагностических endpoint-ов
    \cite{grpc_docs,spring_boot_docs,spring_jdbc_docs,flyway_docs,redis_docs,micrometer_docs}.
\end{enumerate}

Использование Spring JDBC вместо ORM является сознательным решением. Для данной
предметной области критичны контролируемые транзакционные границы, прозрачные
SQL-запросы, явные индексы и предсказуемое поведение при пакетной обработке
очередей и журналов. Поэтому взаимодействие с реляционной базой данных оформлено
через миграции и hand-written SQL, а не через автоматическую генерацию запросов.

\subsection{Моделирование реляционных данных}

\subsubsection{Схема фасада}

Фасад использует схему nf\_fac, в которой сосредоточены как доменные сущности
события уведомления, так и технические записи интеграции. Основные таблицы:
\begin{enumerate}
    \item notification\_event --- корневая сущность уведомительного события;
    \item event\_audience --- описание типа аудитории и её параметров;
    \item event\_recipient --- явный список получателей для EXPLICIT-аудитории;
    \item dispatch --- экземпляр запуска доставки;
    \item dispatch\_target --- снимок получателей конкретного запуска;
    \item outbox\_message --- выходящий журнал интеграционных сообщений.
\end{enumerate}

В таблице notification\_event хранятся идентификатор клиента, ключ идемпотентности,
идентификатор шаблона, версия, приоритет, предпочитаемый канал, стратегия отправки,
статус и JSON-представление полезной нагрузки. На уровне БД задана уникальность по
паре client\_id + idempotency\_key, что обеспечивает защиту от повторного создания
одного и того же события.

Таблица dispatch логически отделяет бизнес-событие от факта его запуска в канал.
Это особенно важно для отложенной отправки и повторных запусков по одной и той же
сущности события. Уникальность event\_id + idempotency\_key не допускает дублей
запусков при сетевых повторах или повторном вызове клиента.

\subsubsection{Схемы канальных сервисов}

Почтовый сервис использует схему nf\_mail, SMS-сервис --- nf\_sms. Несмотря на
различие каналов, структура их хранения одинакова по роли таблиц:
\begin{enumerate}
    \item consumer\_inbox\_message --- входящий журнал Kafka-сообщений;
    \item mail\_delivery или sms\_delivery --- таблица фактических доставок;
    \item mail\_delivery\_attempt или sms\_delivery\_attempt --- история попыток;
    \item outbox\_message --- статусные сообщения для сервиса истории.
\end{enumerate}

В почтовом сервисе дополнительно присутствует таблица recipient\_mail\_settings,
используемая как служебный слой хранения настроек отправки. Канальные таблицы
содержат ключи dispatch\_id, event\_id, client\_id, recipient\_id, тип канала,
статус, payload, идентификатор шаблона, версию шаблона, счётчик попыток,
следующий момент отправки и последнюю ошибку.

Такая модель даёт несколько важных эксплуатационных свойств:
\begin{enumerate}
    \item можно безопасно возобновить обработку после перезапуска сервиса;
    \item каждая повторная попытка фиксируется отдельно;
    \item состояние канального контура можно анализировать независимо от фасада;
    \item история по получателю и каналу строится из фактических, а не вычисляемых состояний.
\end{enumerate}

\subsubsection{Схемы шаблонов, планировщика и истории}

Реестр шаблонов использует схему nf\_tpl с таблицами template и
template\_version. Таблица template хранит идентичность шаблона и номер
активной версии, а template\_version --- движок рендеринга, JSON-массив
канальных представлений и время создания версии.

Планировщик использует схему nf\_sched и таблицу scheduled\_delivery\_task,
в которой хранятся входящие отложенные сообщения, тип агрегата, идентификатор,
payload, заголовки, время исходного создания, состояние задачи и следующая
повторная попытка публикации.

Сервис истории использует схему nf\_hist с таблицей delivery\_history. В ней
сохраняются событие, запуск, получатель, канал, статус, причина, номер попытки,
времена возникновения и служебный outbox\_id, по которому применяется
идемпотентная вставка on conflict do nothing.

\subsection{Модель профиля получателя в Redis}

Сервис profile-consent не хранит профиль в реляционной модели. Он использует
Redis-хеш по ключу с префиксом profile-consent:recipient: и читает из него
следующие атрибуты:
\begin{enumerate}
    \item active;
    \item preferred\_channel;
    \item email.enabled, email.blacklisted, email.destination;
    \item sms.enabled, sms.blacklisted, sms.destination;
    \item push.enabled, push.blacklisted, push.destination;
    \item updated\_at.
\end{enumerate}

На выходе эти поля собираются в объект RecipientProfile, содержащий карту
Channel -> ChannelConsent. Важно, что Redis-модель остаётся внутренней деталью
profile-consent; остальные сервисы работают только с gRPC-ответами и не зависят
от конкретных имён полей, ключей и формата хранения.

\subsection{Контракты gRPC}

\subsubsection{Контракт фасада}

Сервис NotificationFacade реализует три группы операций:
\begin{enumerate}
    \item управление событием: CreateNotificationEvent, UpdateNotificationEvent, CancelNotificationEvent, GetNotificationEvent, ListNotificationEvents;
    \item управление аудиторией: SetAudience, AddRecipients, RemoveRecipients, GetAudience;
    \item управление запуском доставки: TriggerDispatch, GetDispatch, ListDispatches.
\end{enumerate}

Такое разбиение отделяет создание и редактирование события от факта запуска
массовой обработки. С точки зрения проектирования API это существенно лучше, чем
объединять всю логику в одну команду отправки, так как клиент может независимо
создавать сущность, настраивать аудиторию и инициировать доставку.

\subsubsection{Контракт реестра шаблонов}

Сервис TemplateRegistry предоставляет методы CreateTemplate, UpdateTemplate,
GetTemplate, ListTemplates и RenderPreview. Реестр шаблонов используется в
двух режимах:
\begin{enumerate}
    \item как справочник шаблонов и их версий;
    \item как сервис предварительного рендеринга содержимого для выбранного канала.
\end{enumerate}

Метод RenderPreview принимает template\_id, версию, канал и payload, после чего
находит соответствующий канал в template\_version.contents и возвращает
отрендеренные subject и body.

\subsubsection{Контракты профиля получателя и истории}

Сервис ProfileConsentService реализует методы GetRecipientProfile,
BatchGetRecipientProfiles и CheckRecipientChannel. Сервис истории
NotificationHistoryService реализует ListClientHistory,
GetRecipientDeliverySummary и BatchGetRecipientDeliverySummaries.

Пара этих сервисов образует инфраструктурный слой принятия решения о допустимости
доставки. profile-consent отвечает на вопрос «можно ли доставлять по каналу и
какой destination использовать», а history-writer отвечает на вопрос «не
превышена ли допустимая частота доставки в заданном окне времени».

\subsection{Реализация рендеринга шаблонов}

Фактическая логика выбора шаблона и подстановки параметров устроена следующим образом.
\begin{enumerate}
    \item Фасад вызывает resolveDeliveryPayload(...).
    \item Если во входном payload уже есть subject и один из текстовых атрибутов
    (body, text, message, content), встроенный контент используется без
    обращения к template-registry.
    \item Если встроенного контента нет, а интеграция включена, фасад вызывает
    RenderPreview.
    \item Реестр шаблонов выбирает версию шаблона, находит контент по каналу и
    прогоняет subject и body через движок рендеринга.
    \item Полученные subject/body возвращаются в фасад и добавляются в общий payload события.
\end{enumerate}

В проекте реализованы два движка рендеринга:
\begin{enumerate}
    \item TEMPLATE\_ENGINE\_HANDLEBARS --- реальный Handlebars через библиотеку com.github.jknack.handlebars;
    \item TEMPLATE\_ENGINE\_FREEMARKER --- упрощённый рендерер SimpleTemplateRenderer, который выполняет простую замену маркеров \{key\} и \{\{key\}\}, но не реализует полный набор возможностей FreeMarker.
\end{enumerate}

\assumption{Текущий enum TEMPLATE\_ENGINE\_FREEMARKER в кодовой базе не означает
подключение полноценного движка Apache FreeMarker. В текущей реализации это
упрощённый шаблонизатор строк. Если в окончательной версии ВКР требуется
описывать именно FreeMarker как промышленный движок, кодовую реализацию нужно
доработать отдельно.}

Использование предварительного рендера в фасаде принципиально означает, что
подстановка параметров может отработать и до TriggerDispatch. Если subject/body
были внедрены при создании события, то sender-сервисам уже не нужен повторный
вызов реестра шаблонов: они читают готовый payload и отправляют его дальше.

\subsection{Реализация фасадного пайплайна}

\subsubsection{Создание события}

Метод create(...) в JdbcNotificationFacadeUseCase выполняет следующие шаги:
\begin{enumerate}
    \item проверяет дубликат по client\_id + idempotency\_key;
    \item вычисляет scheduledAt и начальный статус по стратегии;
    \item вызывает resolveDeliveryPayload(...);
    \item вставляет запись в notification\_event;
    \item сохраняет аудиторию и, при явном списке, получателей;
    \item записывает в outbox событие EventCreated.
\end{enumerate}

Важно, что публикация EventCreated выполняется даже до TriggerDispatch.
Именно поэтому sender-сервисы могут реагировать на факт создания события и
строить собственные записи планирования отдельно от запуска канальной dispatch-волны.

\subsubsection{Запуск доставки}

Метод triggerDispatch(...) создаёт запись dispatch, при необходимости
снимает snapshot явной аудитории в dispatch\_target и затем вызывает
enqueueChannelDispatch(...). В зависимости от preferred\_channel создаётся
outbox-сообщение одного из двух типов:
\begin{enumerate}
    \item mail\_dispatch / MailDispatchRequested;
    \item sms\_dispatch / SmsDispatchRequested.
\end{enumerate}

Маршрутизация Kafka-топика далее выполняется в OutboxRelayService.topicFor(...).
Для mail\_dispatch дополнительно анализируется planned\_send\_at: если время
отправки лежит в будущем, сообщение уходит в notification.mail.dispatches.scheduled,
иначе сразу в notification.mail.dispatches.

\subsection{Реализация sender-сервисов}

\subsubsection{Почтовый контур}

Почтовый сервис содержит два Kafka-консьюмера: один принимает
notification.facade.events, второй --- notification.mail.dispatches.
Оба консьюмера не создают доставку напрямую, а сохраняют сообщение во входящий
журнал nf\_mail.consumer\_inbox\_message.

Далее работают два workflow-компонента:
\begin{enumerate}
    \item NotificationEventWorkflow обрабатывает EventCreated;
    \item MailDispatchWorkflow обрабатывает MailDispatchRequested и собственно отправку mail\_delivery.
\end{enumerate}

На этапе планирования используется JdbcRecipientDeliveryPolicyEvaluator. Он
сначала вызывает profile-consent.checkRecipientChannel(recipientId, CHANNEL\_EMAIL),
а затем, если включена проверка истории, делает
history-writer.getRecipientDeliverySummary(...). При недоступности истории
почтовый сервис работает в режиме fail-open и не блокирует доставку только из-за
ошибки чтения агрегата истории.

Шлюз SpringMailGateway формирует SimpleMailMessage из payload. Он берёт
subject из subject или title, а текст --- из body, text, message
или content. Если включён режим log-only, фактическая отправка заменяется
логированием и доставка считается успешной в рамках тестового контура. Если
режим отключён, используется JavaMailSender.

\subsubsection{SMS-контур}

SMS-сервис повторяет общую схему: принимает события и dispatch-сообщения, пишет их
во входящий журнал nf\_sms.consumer\_inbox\_message, затем отдельные workflow-ы
создают записи в sms\_delivery и выполняют отправку.

Отличие состоит в двух деталях.
\begin{enumerate}
    \item Планирование в SmsDeliveryPlanService выполняется пакетно по списку получателей.
    \item Перед самой отправкой LoggingSmsGateway дополнительно прогоняет batch
    через SmsProviderGuardService, который ещё раз вызывает profile-consent.
\end{enumerate}

Эта дополнительная SMS-проверка полезна как защитный барьер перед шлюзом, однако
она увеличивает число профильных вызовов и должна учитываться при анализе
нагрузочного профиля системы.

\subsection{Реализация сервиса истории}

Сервис history-writer имеет два канала поступления данных:
\begin{enumerate}
    \item Kafka-консьюмер DeliveryHistoryConsumer, читающий
    notification.mail.delivery-statuses и notification.sms.delivery-statuses;
    \item gRPC-сервис чтения истории и агрегированных summary.
\end{enumerate}

Kafka-слой выполняет только нормализацию входящего статусного события и запись в
delivery\_history. gRPC-слой не вмешивается в доставку, а предоставляет историю
клиентам и другим сервисам. Метод GetRecipientDeliverySummary валидирует
recipient\_id, нормализует lookback\_hours и вычисляет сводку по каналу в
заданном временном окне.

\subsection{Конфигурация и эксплуатационные ограничения текущей реализации}

Текущая кодовая база содержит несколько особенностей, которые необходимо явно
зафиксировать в отчёте.
\begin{enumerate}
    \item В mail- и SMS-сервисах присутствуют режимы тестовой отправки, когда внешнее взаимодействие заменяется логированием.
    \item Для части gRPC-клиентов и Kafka-подключений в демонстрационном профиле используется plaintext.
    \item В профиле получателя предусмотрен канал push, но отдельный push-sender в репозитории отсутствует.
    \item Топик notification.facade.dispatches публикуется фасадом, однако в текущей реализации не участвует в прикладной обработке sender-сервисов.
\end{enumerate}

Эти особенности не делают систему некорректной, но определяют границы того, что
можно честно утверждать о текущем программном результате.

\subsection{Выводы по разделу}

Реализация подтверждает жизнеспособность выбранной архитектуры: все критические
сущности отражены в явной модели данных, межсервисное взаимодействие описано
protobuf-контрактами, а доставка выполняется через воспроизводимый Kafka-пайплайн
с outbox/inbox-механизмами. Особое значение имеют три факта: рендер шаблона
может происходить уже на этапе создания события, sender-сервисы работают с
устойчиво сохранёнными входящими сообщениями, а почтовый и SMS-контуры имеют
разную глубину проверок получателя. Эти различия принципиальны для корректного
понимания поведения системы под нагрузкой и при восстановлении после сбоев.
